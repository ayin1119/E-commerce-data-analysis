# RFM 分析代码封装优化与时间复杂度审查报告

## 1. 报告目的

本报告用于说明 RFM 分析代码从旧版脚本到封装版代码的优化过程，并对封装后的代码进行时间复杂度审查。

旧版代码将数据读取、数据合并、字段转换、RFM 打分、用户分层、结果汇总和文件保存全部写在同一个脚本中。封装后代码将这些步骤拆分为多个函数，并通过统一入口函数 `run_rfm_analysis(config)` 执行完整流程。这样做的目的不是改变算法结果，而是提高代码结构清晰度、复用性、可维护性和异常检查能力。

## 2. 审查对象

| 文件                | 说明                                            |
| ------------------- | ----------------------------------------------- |
| `rfm_old.py`      | 旧版 RFM 分析代码，所有流程写在同一个脚本中     |
| `rfm_pipeline.py` | 封装后的 RFM 分析代码，将不同处理步骤拆分为函数 |

## 3. 旧版代码审查

旧版代码能够完成 RFM 用户分层分析，但整体属于过程式脚本写法。代码从上到下依次执行，路径配置、数据读取、数据处理、用户分层和结果保存都集中在一个文件中。

旧版代码的主要流程如下：

```text
1. 设置三个输入文件路径和输出文件夹
2. 读取三张 CSV 表
3. 按 user_id 合并三张表
4. 将需要计算的字段转换为数值类型
5. 拆分购买用户和未购买用户
6. 对购买用户计算 R_score、F_score、M_score
7. 根据平均分判断 R/F/M 高低
8. 根据 R/F/M 高低组合划分用户类型
9. 合并购买用户和未购买用户
10. 生成用户类型汇总表
11. 保存用户级结果和汇总结果
```

旧版代码的主要问题不是计算逻辑错误，而是工程结构不够清晰。具体表现为：第一，输入路径、输出路径和分析逻辑写在一起，如果后续更换文件路径，需要到主代码中修改，容易误改核心逻辑。第二，所有代码集中在一个脚本中，代码较长，不方便定位每一步的作用。第三，R/F/M 打分代码分别重复书写，虽然可以运行，但复用性较弱。第四，旧版代码缺少对关键字段的统一检查，如果输入文件缺少必要字段，可能要等运行到中间步骤才报错，排查不够直观。

## 4. 封装后代码审查

封装后的 `rfm_pipeline.py` 将原来集中在一个脚本中的流程拆分成多个函数。每个函数只负责一个相对独立的任务，使代码结构更加清晰。

封装后的主要函数如下：

| 函数名                             | 作用                            |
| ---------------------------------- | ------------------------------- |
| `load_data(config)`              | 根据配置读取三张 CSV 表         |
| `merge_tables(...)`              | 按`user_id` 合并三张用户表    |
| `convert_numeric_columns(...)`   | 将指定字段转换为数值类型        |
| `check_required_columns(...)`    | 检查关键字段是否存在            |
| `split_purchase_users(...)`      | 拆分购买用户和未购买用户        |
| `make_score_by_quantile(...)`    | 通用的分位数打分函数            |
| `add_rfm_scores(...)`            | 计算 R/F/M 三个分数             |
| `add_rfm_levels(...)`            | 根据平均分生成高低标签          |
| `add_rfm_type(...)`              | 根据 R/F/M 高低组合生成用户类型 |
| `handle_non_purchase_users(...)` | 处理未购买用户                  |
| `build_rfm_result(...)`          | 生成最终用户级 RFM 结果表       |
| `build_rfm_summary(...)`         | 生成用户类型汇总表              |
| `save_results(...)`              | 保存分析结果                    |
| `run_rfm_analysis(config)`       | 统一运行入口                    |

封装后的代码保留了旧版 RFM 分析的核心业务逻辑，但对代码组织方式进行了优化。旧版代码是“一整段流程直接执行”，封装后代码变成了“多个函数分工完成，再由统一入口调用”。这种修改使代码更容易阅读，也更方便后续检查和复用。

## 5. 时间复杂度审查

假设最终用户数量为 `n`，购买用户数量为 `p`，并且 `p <= n`。

### 5.1 读取数据

读取三张 CSV 表需要扫描文件中的每一行数据，因此时间复杂度可以近似看作：

```text
O(n)
```

### 5.2 合并数据

三张表按 `user_id` 进行合并。pandas 的 `merge()` 通常可以近似看作线性合并，因此整体可以近似看作：

```text
O(n)
```

### 5.3 字段类型转换

数值字段转换需要逐行扫描指定字段，因此时间复杂度为：

```text
O(n)
```

### 5.4 拆分购买用户和未购买用户

根据 `purchase_count` 判断用户是否购买，需要对每个用户判断一次，因此时间复杂度为：

```text
O(n)
```

### 5.5 RFM 打分

RFM 打分使用 `rank()` 和 `qcut()` 进行排名和分位数分箱。排名和分箱通常涉及排序，因此单个指标的时间复杂度为：

```text
O(p log p)
```

R、F、M 三个指标都需要打分，因此可以写为：

```text
3 × O(p log p)
```

时间复杂度分析中通常忽略常数，因此仍然记为：

```text
O(p log p)
```

由于 `p <= n`，所以也可以近似写为：

```text
O(n log n)
```

### 5.6 生成高低标签和用户类型

根据 R/F/M 分数生成高低标签，并根据高低组合生成用户类型，都需要对购买用户进行一次判断，因此时间复杂度为：

```text
O(p)
```

### 5.7 汇总统计

按照用户类型进行 `groupby()` 汇总。虽然用户类型数量很少，但程序仍然需要扫描用户结果表，因此时间复杂度为：

```text
O(n)
```

### 5.8 保存结果

保存 CSV 需要将结果逐行写出，因此时间复杂度为：

```text
O(n)
```

## 6. 总体时间复杂度结论

将所有步骤合并后，可以得到：

```text
读取数据：O(n)
合并数据：O(n)
字段转换：O(n)
拆分用户：O(n)
RFM 打分：O(n log n)
用户分层：O(n)
汇总统计：O(n)
保存结果：O(n)
```

其中增长最快的是 RFM 打分阶段的 `O(n log n)`，因此整个程序的总体时间复杂度为：

```text
O(n log n)
```

程序需要保存合并后的用户表、购买用户表、未购买用户表、RFM 分数、用户标签和最终结果，因此空间复杂度为：

```text
O(n)
```

## 7. 封装前后复杂度对比

| 对比项             | 旧版代码       | 封装后代码     |
| ------------------ | -------------- | -------------- |
| 总体时间复杂度     | `O(n log n)` | `O(n log n)` |
| 空间复杂度         | `O(n)`       | `O(n)`       |
| 主要耗时步骤       | RFM 分位数打分 | RFM 分位数打分 |
| 是否改变算法复杂度 | 否             | 否             |
| 是否提升代码结构   | 否             | 是             |
| 是否减少重复代码   | 否             | 是             |
| 是否方便复用       | 一般           | 更方便         |
| 是否方便排查错误   | 一般           | 更方便         |

封装后代码没有改变 RFM 分析的核心算法，因此总体时间复杂度仍然是 `O(n log n)`。但是封装后代码将不同步骤拆分为函数，减少了重复代码，增强了字段检查能力，也让主流程更加清晰。因此本次优化属于代码结构优化、可维护性优化和稳定性优化，而不是算法复杂度阶数上的优化。

## 9. 最终结论

本次代码优化主要完成了对旧版 RFM 分析脚本的封装改造。旧版代码虽然可以完成 RFM 用户分层，但所有逻辑集中在一个脚本中，代码结构较长，路径配置和分析逻辑耦合在一起，不利于维护和复用。封装后代码将读取数据、合并数据、字段转换、字段检查、用户拆分、RFM 打分、用户分层、结果汇总和保存结果分别封装为函数，并通过 `run_rfm_analysis(config)` 作为统一接口运行完整流程。

从时间复杂度角度看，读取、合并、转换、筛选、分层、汇总和保存等步骤主要是线性操作，时间复杂度为 `O(n)`；RFM 打分使用排名和分位数分箱，涉及排序，因此时间复杂度为 `O(n log n)`。由于该步骤是整个流程中增长最快的部分，所以程序整体时间复杂度为 `O(n log n)`，空间复杂度为 `O(n)`。

因此，封装后的代码在总体算法复杂度上与旧版保持一致，但在代码清晰度、复用性、可维护性、稳定性和错误排查便利性方面都有明显提升。
