# RFM 各层级 Top10 商品类别分析

# 1. 报告背景

本报告围绕“RFM 各层级 Top10 商品类别分析”进行代码规范化改造和时间复杂度优化说明。原始分析目标是结合用户行为数据表和 RFM 用户分层结果表，统计不同 RFM 用户层级下购买次数最高的 Top10 商品类别，并计算对应的购买次数、购买用户数、购买次数占比和购买用户占比。

本次修改遵循四个要求：第一，进行接口定义，让外部只需要调用一个统一函数即可完成完整分析；第二，进行代码封装，将读取数据、分块处理、统计汇总、Top10 筛选、结果保存拆分为独立函数；第三，设置配置参数文件，将路径、字段名、购买行为编号、TopK 数量等参数集中管理；第四，计算并说明时间复杂度，同时尽可能从算法层面降低时间复杂度。

## 2. 修改目标

本次修改的目标不是改变业务分析结果，而是在保持“统计各 RFM 层级购买 Top10 商品类别”这一目标不变的基础上，提高代码结构清晰度、复用性、运行稳定性和大数据量处理能力。

具体目标包括：

| 修改目标       | 说明                                                             |
| -------------- | ---------------------------------------------------------------- |
| 配置参数文件   | 将输入路径、输出路径、字段名、TopK 数量等参数放入`config.py`   |
| 接口定义       | 使用`run_rfm_top10_category_analysis(config)` 作为统一运行接口 |
| 代码封装       | 将完整流程拆分为多个函数，每个函数只负责一个步骤                 |
| 时间复杂度优化 | 将“全量排序取 Top10”优化为“每组直接取 TopK”                  |
| 内存优化       | 使用`chunksize` 分块读取原始行为数据，避免一次性读取完整大表   |
| 可维护性优化   | 路径、字段、参数统一管理，后续修改不需要改核心代码               |

## 3. 修改后的项目结构

修改后的项目建议拆分为三个 Python 文件：

```text
rfm_top10_category_project/
│
├── config.py
├── rfm_top10_category_pipeline.py
└── run_rfm_top10_category.py
```

三个文件的职责如下：

| 文件                               | 作用                                                                            |
| ---------------------------------- | ------------------------------------------------------------------------------- |
| `config.py`                      | 配置参数文件，负责保存输入路径、输出路径、字段名、购买行为编号、TopK 数量等参数 |
| `rfm_top10_category_pipeline.py` | 核心代码封装文件，负责读取数据、分块处理、统计汇总、TopK 筛选和保存结果         |
| `run_rfm_top10_category.py`      | 运行入口文件，负责导入配置并调用统一接口运行程序                                |

这样拆分后，代码结构更清晰。初学者只需要知道：改路径去 `config.py`，看核心逻辑去 `rfm_top10_category_pipeline.py`，运行程序用 `run_rfm_top10_category.py`。

## 4. 业务处理流程

修改后的程序整体流程如下：

```text
读取配置参数
      ↓
读取 RFM 用户分层结果表
      ↓
分块读取原始行为数据 data_min
      ↓
只保留 behavior_type = 4 的购买行为
      ↓
按 user_id 合并 RFM 用户层级
      ↓
按 rfm_type + item_category 统计购买次数
      ↓
统计每个类别的购买用户数
      ↓
计算购买次数占比和购买用户占比
      ↓
每个 RFM 层级取 Top10 类别
      ↓
保存结果 CSV
```

最终输出结果表字段包括：

| 字段                         | 含义                                     |
| ---------------------------- | ---------------------------------------- |
| `rfm_type`                 | RFM 用户层级                             |
| `category_rank`            | 商品类别在该 RFM 层级下的排名            |
| `item_category`            | 商品类别                                 |
| `purchase_count`           | 该类别购买次数                           |
| `purchase_user_count`      | 买过该类别的用户数                       |
| `rfm_total_purchase_count` | 该 RFM 层级总购买次数                    |
| `rfm_purchase_user_count`  | 该 RFM 层级购买用户总数                  |
| `purchase_count_rate`      | 该类别购买次数占该层级总购买次数的比例   |
| `purchase_user_rate`       | 该类别购买用户数占该层级购买用户数的比例 |

## 5. 时间复杂度优化前分析

设：

```text
N：原始行为数据总行数
P：购买行为数据行数，P <= N
G：RFM 层级与商品类别的组合数量
K：每个 RFM 层级需要取出的商品类别数量，本项目中 K = 10
U：用户-层级-类别去重后的组合数量
```

原始思路一般是先得到所有 `rfm_type + item_category` 的统计结果，然后对全部统计结果进行完整排序，再取每个 RFM 层级下排名前 10 的商品类别。

原始流程的主要时间复杂度如下：

| 步骤              | 时间复杂度     | 说明                                        |
| ----------------- | -------------- | ------------------------------------------- |
| 读取原始行为数据  | `O(N)`       | 需要扫描所有行为记录                        |
| 筛选购买行为      | `O(N)`       | 判断每一行是否为购买行为                    |
| 合并 RFM 用户层级 | `O(P)`       | 只对购买行为数据进行合并                    |
| 分组统计购买次数  | `O(P)`       | 按`rfm_type + item_category` 统计购买次数 |
| 完整排序          | `O(G log G)` | 对所有类别统计结果进行完整排序              |

因此，优化前整体时间复杂度可以表示为：

```text
O(N + P + G log G)
```

其中最值得优化的是 `G log G`。因为如果只是为了取 Top10，没有必要对所有类别进行完整排序。

## 6. 时间复杂度优化方案

本次优化的核心是将“完整排序后取 Top10”改为“每个 RFM 层级直接取 TopK”。

原始做法是：

```text
先对全部 G 个组合完整排序
再取每个层级前 10 个
```

优化后做法是：

```text
每个 RFM 层级内部直接取购买次数最高的前 K 个
```

核心代码为：

```python
top_index = (
    category_summary
    .groupby(rfm_type_col)["purchase_count"]
    .nlargest(top_k)
    .index
    .get_level_values(-1)
)
```

这段代码的作用是：按照 `rfm_type` 分组，在每个 RFM 层级内部，只选出 `purchase_count` 最大的前 `top_k` 个类别。

优化前 Top10 部分复杂度为：

```text
O(G log G)
```

优化后 TopK 部分复杂度为：

```text
O(G log K)
```

由于本项目中 `K = 10`，是固定常数，因此 `log K` 可以近似看作常数。这样 TopK 部分可以近似看作：

```text
O(G)
```

## 7. 优化后的时间复杂度

优化后，主要流程的时间复杂度如下：

| 步骤                 | 时间复杂度     | 说明                                     |
| -------------------- | -------------- | ---------------------------------------- |
| 分块读取原始行为数据 | `O(N)`       | 虽然分块读取，但所有数据仍然需要扫描一遍 |
| 筛选购买行为         | `O(N)`       | 判断每一行是否为购买行为                 |
| 合并 RFM 用户层级    | `O(P)`       | 只对购买行为进行合并                     |
| 分组统计购买次数     | `O(P)`       | 按 RFM 层级和商品类别统计                |
| 统计购买用户数       | `O(U)`       | 对用户-层级-类别组合去重后统计           |
| TopK 选择            | `O(G log K)` | 每个层级只取前 K 个商品类别              |
| 保存结果             | `O(G)`       | 保存最终统计结果                         |

所以优化后的整体时间复杂度为：

```text
O(N + P + U + G log K)
```

因为 `K = 10` 是固定常数，所以可以近似为：

```text
O(N + P + U + G)
```

又因为购买行为数量 `P` 不会超过原始总行为数量 `N`，所以可以进一步近似理解为：

```text
O(N + U + G)
```

在业务报告中，为了便于理解，也可以写成：

```text
优化后整体复杂度接近 O(N)
```

## 8. 空间复杂度分析

空间复杂度主要来自以下中间数据：

| 数据                 | 说明                                            |
| -------------------- | ----------------------------------------------- |
| RFM 用户分层表       | 保存`user_id` 与 `rfm_type` 的对应关系      |
| 购买次数统计表       | 保存每个`rfm_type + item_category` 的购买次数 |
| 用户-层级-类别去重表 | 用于计算购买用户数                              |
| 最终 Top10 结果表    | 保存最终输出结果                                |

由于程序采用 `chunksize` 分块读取原始行为数据，因此不会一次性把完整 `data_min` 表全部读入内存。空间复杂度主要取决于统计结果规模，而不是原始行为数据总规模。

空间复杂度可以表示为：

```text
O(G + U)
```

其中，`G` 是 RFM 层级与商品类别的组合数量，`U` 是用户-层级-类别去重后的组合数量。

## 9. 优化前后对比

| 对比项     | 优化前                                   | 优化后                                          | 优化效果                                 |
| ---------- | ---------------------------------------- | ----------------------------------------------- | ---------------------------------------- |
| 参数管理   | 路径、字段、输出文件可能直接写在主代码里 | 使用`config.py` 集中管理                      | 更容易修改和维护                         |
| 代码结构   | 从上到下顺序执行，逻辑集中在一起         | 拆分为多个函数                                  | 可读性和复用性提升                       |
| 运行接口   | 没有统一入口                             | 使用`run_rfm_top10_category_analysis(config)` | 调用更简单                               |
| 大表读取   | 可能一次性读取完整数据                   | 使用`chunksize` 分块读取                      | 降低内存压力                             |
| 行为筛选   | 读取后再整体处理                         | 分块时直接筛选购买行为                          | 减少后续处理数据量                       |
| Top10 计算 | 全量排序后取前 10                        | 每组直接取 TopK                                 | 时间复杂度降低                           |
| 时间复杂度 | `O(N + P + G log G)`                   | `O(N + P + U + G log K)`                      | Top10 部分由`G log G` 降为 `G log K` |
| 空间复杂度 | 可能接近`O(N)`                         | `O(G + U)`                                    | 内存占用更合理                           |

## 10. 最终结论

本次修改完成了接口定义、代码封装、配置参数文件设计和时间复杂度优化。修改后的项目由 `config.py`、`rfm_top10_category_pipeline.py` 和 `run_rfm_top10_category.py` 三个文件组成。其中，`config.py` 负责统一管理参数，`rfm_top10_category_pipeline.py` 负责核心分析流程，`run_rfm_top10_category.py` 负责运行程序。

在时间复杂度方面，原始思路需要对所有 RFM 层级和商品类别组合进行完整排序，Top10 部分复杂度为 `O(G log G)`。优化后使用 TopK 思路，每个 RFM 层级只选出购买次数最高的前 K 个商品类别，Top10 部分复杂度优化为 `O(G log K)`。由于本项目中 `K = 10` 是固定常数，因此该部分可以近似看作 `O(G)`。整体复杂度由 `O(N + P + G log G)` 优化为 `O(N + P + U + G log K)`，整体运行趋势更接近线性。

在空间复杂度方面，修改后使用分块读取原始行为数据，不再一次性加载完整大表，主要保存购买统计结果和用户-类别去重结果，空间复杂度为 `O(G + U)`。因此，本次修改不仅满足了接口定义、代码封装和配置文件管理的要求，也在 Top10 筛选环节实现了明确的时间复杂度优化。
