# RFM 各层级 Top100 商品购买偏好分析

## 1. 报告背景

本报告围绕“RFM 各层级 Top100 商品购买偏好分析”进行代码规范化修改与时间复杂度优化说明。原始任务是结合用户行为数据表 `data_min.csv` 和 RFM 用户分层结果表，统计不同 RFM 用户层级下购买次数最高的前 100 个商品，并计算对应的购买次数、购买用户数、购买次数占比和购买用户占比。

## 2. 修改目标

| 修改目标       | 说明                                                                                     |
| -------------- | ---------------------------------------------------------------------------------------- |
| 配置参数文件   | 将输入路径、输出路径、字段名、购买行为编号、TopK 数量等参数集中放入`config.py`         |
| 接口定义       | 定义统一接口`run_rfm_top100_item_analysis(config)`，外部只需调用该函数即可完成完整分析 |
| 代码封装       | 将读取 RFM、分块处理购买行为、统计汇总、Top100 筛选、结果保存拆分成独立函数              |
| 时间复杂度优化 | 将“全量排序取 Top100”优化为“每个 RFM 层级直接取 TopK”                                |
| 统计准确性优化 | 通过统一保存用户-层级-商品关系并整体去重，避免购买用户数重复计算                         |
| 内存优化       | 使用`chunksize` 分块读取原始行为数据，避免一次性读取完整大表                           |

## 3. 修改后的项目结构

修改后的项目建议拆分为三个 Python 文件：

```text
rfm_top100_item_project/
│
├── config.py
├── rfm_top100_item_pipeline.py
└── run_rfm_top100_item.py
```

三个文件的职责如下：

| 文件                            | 作用                                                                      |
| ------------------------------- | ------------------------------------------------------------------------- |
| `config.py`                   | 配置参数文件，保存路径、字段名、购买行为编号、TopK 数量、输出文件名等参数 |
| `rfm_top100_item_pipeline.py` | 核心代码封装文件，包含读取、分块处理、统计汇总、TopK 筛选和保存结果等函数 |
| `run_rfm_top100_item.py`      | 运行入口文件，负责导入配置并调用统一接口运行程序                          |

这种结构的好处是职责清晰。修改路径时只改 `config.py`，查看核心逻辑时只看 `rfm_top100_item_pipeline.py`，运行程序时只执行 `run_rfm_top100_item.py`。

## 4. 修改后的业务流程

修改后的程序整体流程如下：

```text
读取配置参数
      ↓
读取 RFM 用户分层结果表
      ↓
生成 user_id 到 rfm_type 的映射字典
      ↓
分块读取原始行为数据 data_min
      ↓
只保留 behavior_type = 4 的购买行为
      ↓
使用 map 给购买记录补充 RFM 用户层级
      ↓
按 rfm_type + item_category + item_id 统计购买次数
      ↓
保存 user_id + rfm_type + item_category + item_id 去重关系
      ↓
统一计算购买用户数
      ↓
计算购买次数占比和购买用户占比
      ↓
每个 RFM 层级取 Top100 商品
      ↓
保存结果 CSV
```

rfm_each_level_top100_items.csv

输出字段如下：

其中：

```text
purchase_count_rate = purchase_count / rfm_total_purchase_count
```

```text
purchase_user_rate = purchase_user_count / rfm_purchase_user_count
```

## 6. 时间复杂度符号定义

为了分析时间复杂度，先定义以下符号：

```text
N：原始行为数据 data_min 的总行数
P：购买行为数据行数，即 behavior_type = 4 的行数
R：RFM 用户分层结果表中的用户数量
G：rfm_type + item_category + item_id 的组合数量
U：user_id + rfm_type + item_category + item_id 的去重组合数量
K：每个 RFM 层级取前 K 个商品，本项目中 K = 100
```

其中：

```text
P <= N
```

因为购买行为只是全部行为中的一部分。

## 7. 原始时间复杂度分析

原始做法通常是：

```text
读取完整行为数据
筛选购买行为
合并 RFM 用户分层表
按 rfm_type + item_category + item_id 分组统计
计算购买用户数和占比
对所有商品组合完整排序或排名
取每个 RFM 层级前 100 个商品
```

各步骤时间复杂度如下：

| 步骤             | 时间复杂度      | 说明                     |
| ---------------- | --------------- | ------------------------ |
| 读取原始行为数据 | `O(N)`        | 需要扫描所有行为记录     |
| 筛选购买行为     | `O(N)`        | 判断每一行是否为购买行为 |
| 合并 RFM 分层    | `O(P)` 或更高 | 对购买行为补充用户层级   |
| 分组统计购买次数 | `O(P)`        | 按层级和商品分组         |
| 统计购买用户数   | `O(U)`        | 需要对用户-商品关系去重  |
| 全量排序或排名   | `O(G log G)`  | 对所有商品组合完整排序   |
| 保存结果         | `O(G)`        | 将结果写出               |

因此，原始整体时间复杂度可以表示为：

```text
O(N + P + U + G log G)
```

其中最明显的优化点是最后的全量排序。因为本项目只需要每个 RFM 层级前 100 个商品，不需要对全部商品完整排序。

## 8. 时间复杂度优化方案

本次优化主要包括三个方面。

第一，将每个分块中的大表 `merge` 改为 `map` 映射。先将 RFM 用户分层结果表转换为字典：

```python
rfm_map = dict(zip(rfm[user_col], rfm[rfm_type_col]))
```

然后在每个购买行为分块中使用：

```python
purchase_chunk[rfm_type_col] = (
    purchase_chunk[user_col]
    .map(rfm_map)
    .fillna(unknown_user_label)
)
```

这样做的好处是不用每次都进行 DataFrame 级别的大表合并，而是直接根据 `user_id` 查找对应的 RFM 层级。理论上仍然需要处理购买行为数据，但实际运行更轻，代码也更简单。

第二，修正购买用户数统计方式。原始做法如果在每个分块内先计算 `nunique`，再将各分块结果相加，可能导致同一个用户跨分块购买同一商品时被重复统计。修改后保存：

```text
user_id + rfm_type + item_category + item_id
```

的去重关系，最后统一 `drop_duplicates()`，再计算购买用户数。这样可以保证 `purchase_user_count` 更准确。

第三，将全量排序取 Top100 优化为每个 RFM 层级内部直接取 TopK。核心代码为：

```python
top_index = (
    item_summary
    .groupby(rfm_type_col)["purchase_count"]
    .nlargest(top_k)
    .index
    .get_level_values(-1)
)
```

原来的做法是对所有 `G` 个商品组合进行完整排序，复杂度为：

```text
O(G log G)
```

优化后每个 RFM 层级只取前 `K` 个商品，复杂度变为：

```text
O(G log K)
```

本项目中 `K = 100`，是固定值，因此比 `O(G log G)` 更低。

## 9. 优化后的时间复杂度分析

优化后的主要步骤如下：

| 步骤                   | 时间复杂度     | 说明                                      |
| ---------------------- | -------------- | ----------------------------------------- |
| 读取 RFM 用户分层表    | `O(R)`       | 读取用户层级并生成映射字典                |
| 分块读取原始行为数据   | `O(N)`       | 虽然分块读取，但仍需扫描全部数据          |
| 筛选购买行为           | `O(N)`       | 判断每一行是否为购买行为                  |
| 使用 map 映射 RFM 层级 | `O(P)`       | 只对购买行为补充用户层级                  |
| 分组统计购买次数       | `O(P)`       | 按层级和商品统计购买次数                  |
| 用户-商品关系去重      | `O(U)`       | 用于准确计算购买用户数                    |
| TopK 选择              | `O(G log K)` | 每个层级只取前 K 个商品                   |
| 保存 Top100 结果       | `O(K * M)`   | M 为 RFM 层级数量，K 为每层级保留的商品数 |

因此，优化后的整体时间复杂度可以表示为：

```text
O(R + N + P + U + G log K)
```

由于 `P <= N`，可以进一步理解为：

```text
O(R + N + U + G log K)
```

又因为本项目中 `K = 100` 是固定值，`log K` 可以看作较小常数，因此 TopK 部分明显低于全量排序。相比原始的：

```text
O(N + P + U + G log G)
```

优化后变为：

```text
O(R + N + P + U + G log K)
```

主要优化点是：

```text
G log G → G log K
```

## 10. 空间复杂度分析

优化后的程序没有一次性把完整 `data_min` 表读入内存，而是使用 `chunksize` 分块读取。程序主要保存以下中间结果：

| 数据                  | 说明                          |
| --------------------- | ----------------------------- |
| `rfm_map`           | 用户编号到 RFM 层级的映射字典 |
| `purchase_count_df` | 商品购买次数统计表            |
| `user_item_pairs`   | 用户-层级-商品去重关系表      |
| `item_summary`      | 商品购买偏好汇总表            |
| `result`            | 最终 Top100 商品结果表        |

因此空间复杂度主要为：

```text
O(R + G + U)
```

其中，`R` 是 RFM 用户数量，`G` 是 RFM 层级与商品组合数量，`U` 是用户-层级-商品去重组合数量。

因为原始行为数据采用分块读取，不会完整保存在内存中，所以相比一次性读取完整数据表，空间压力更小。

## 11. 优化前后对比

| 对比项          | 优化前                                    | 优化后                                       | 优化效果       |
| --------------- | ----------------------------------------- | -------------------------------------------- | -------------- |
| 参数管理        | 路径、字段名、Top100 可能直接写在主代码中 | 使用`config.py` 统一管理                   | 修改参数更方便 |
| 代码结构        | 所有逻辑集中在 notebook 中                | 拆分为配置、核心封装、运行入口               | 可读性提升     |
| 接口设计        | 没有统一主接口                            | 使用`run_rfm_top100_item_analysis(config)` | 调用更简单     |
| RFM 匹配方式    | 分块中使用`merge` 合并                  | 先建立`rfm_map`，再用 `map` 映射         | 实际运行更轻   |
| 购买用户数统计  | 分块内`nunique` 后相加，可能重复        | 保存用户-商品关系，最后统一去重              | 统计更准确     |
| Top100 获取方式 | 全量排序或排名后取前 100                  | 每组直接`nlargest(top_k)`                  | 时间复杂度降低 |
| 时间复杂度      | `O(N + P + U + G log G)`                | `O(R + N + P + U + G log K)`               | 排序部分降低   |
| 空间复杂度      | 可能接近`O(N)`                          | `O(R + G + U)`                             | 降低内存压力   |

## 12. 最终结论

本次修改完成了 RFM 各层级 Top100 商品购买偏好分析代码的配置参数文件设计、接口定义、代码封装和时间复杂度优化。

在配置参数方面，使用 `config.py` 集中管理输入路径、输出路径、字段名、购买行为编号、未知用户标记、分块大小、TopK 数量和输出文件名。这样后续修改路径或参数时，不需要改动核心代码。

在代码封装方面，将完整分析流程拆分为 `load_rfm_mapping()`、`process_purchase_chunks()`、`build_item_summary()`、`get_top_k_items()`、`save_result()` 和 `run_rfm_top100_item_analysis()` 等函数。每个函数只负责一个步骤，代码结构更加清晰。

在接口定义方面，使用 `run_rfm_top100_item_analysis(config)` 作为统一接口。外部运行文件只需要导入配置并调用该函数，即可完成完整分析流程。

在时间复杂度方面，原始做法需要对所有商品组合进行完整排序或排名，整体复杂度为：

```text
O(N + P + U + G log G)
```

优化后使用每个 RFM 层级内部 TopK 选择，整体复杂度变为：

```text
O(R + N + P + U + G log K)
```

其中 `K = 100` 是固定值，因此相比 `G log G`，`G log K` 明显更低。本次优化的核心是将全量排序改为 TopK 选择，同时通过分块读取减少内存压力，并通过统一去重修正购买用户数统计问题。
