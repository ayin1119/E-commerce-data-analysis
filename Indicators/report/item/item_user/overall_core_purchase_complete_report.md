# 整体核心用户购买率分析

## 1. 报告背景

本报告围绕“整体核心用户购买率分析”进行代码规范化修改与时间复杂度优化说明。该分析的核心目标是计算核心用户在整体购买行为中的贡献情况，即统计全体购买次数、核心用户购买次数、非核心用户购买次数，并计算核心用户购买贡献率和非核心用户购买贡献率。

本项目中的核心用户定义为 RFM 用户分层结果中 `rfm_type = 重要价值用户` 的用户。核心计算公式如下：

```text
核心用户购买贡献率 = 核心用户购买次数 / 全体购买次数
```

## 2. 修改目标

本次修改目标如下：

| 修改目标       | 说明                                                                            |
| -------------- | ------------------------------------------------------------------------------- |
| 配置参数文件   | 将输入路径、输出路径、字段名、核心用户定义、购买行为编号等参数放入`config.py` |
| 接口定义       | 定义统一接口`run_overall_core_purchase_analysis(config)`                      |
| 代码封装       | 将读取核心用户、分块统计购买次数、计算结果、保存结果拆分为独立函数              |
| 时间复杂度计算 | 分析优化前后主要步骤的时间复杂度                                                |
| 内存优化       | 使用`chunksize` 分块读取原始行为数据，避免一次性读取完整大表                  |
| 运行效率优化   | 只读取必要字段，并使用核心用户集合判断，避免整表合并                            |

## 3. 修改后的项目结构

修改后的项目建议拆分为三个 Python 文件：

```text
overall_core_purchase_project/
│
├── config.py
├── overall_core_purchase_pipeline.py

```

三个文件的职责如下：

| 文件                                  | 作用                                                                                           |

| `overall_core_purchase_pipeline.py` | 核心代码封装文件，包含读取核心用户、分块统计购买次数、计算购买率和保存结果等函数               |
| `run_overall_core_purchase.py`      | 运行入口文件，负责导入配置并调用统一接口运行程序                                               |

这种拆分方式可以让代码结构更清晰。以后修改路径只需要改 `config.py`，查看核心逻辑只需要看 `overall_core_purchase_pipeline.py`，运行程序只需要执行 `run_overall_core_purchase.py`。

## 4. 修改后的业务流程

修改后的程序整体流程如下：

```text
读取配置参数

```

## 6. 时间复杂度符号定义

为了分析时间复杂度，先定义以下符号：

```text
N：原始行为数据 data_min 的总行数
P：购买行为数据行数，即 behavior_type = 4 的行数
R：RFM 用户分层结果表中的用户数量
C：核心用户数量
B：单个分块大小，即 chunksize
```

其中：

```text
P <= N
C <= R
```

因为购买行为只是全部行为中的一部分，核心用户也只是 RFM 用户表中的一部分。

## 7. 原始时间复杂度分析

原始做法通常是：

```text
读取完整 data_min 表
读取完整 RFM 用户分层表
筛选核心用户
将行为表和 RFM 表进行合并
筛选购买行为
统计全体购买次数
统计核心用户购买次数
计算核心用户购买率
保存结果
```

各步骤时间复杂度如下：

| 步骤                 | 时间复杂度      | 说明                           |
| -------------------- | --------------- | ------------------------------ |
| 读取完整行为表       | `O(N)`        | 需要读取所有行为记录           |
| 读取 RFM 分层表      | `O(R)`        | 需要读取所有用户分层记录       |
| 合并行为表和 RFM 表  | `O(N)` 或更高 | 对所有行为记录补充用户层级     |
| 筛选购买行为         | `O(N)`        | 判断每一行是否为购买行为       |
| 统计核心用户购买次数 | `O(P)`        | 在购买行为中判断是否为核心用户 |
| 保存结果             | `O(1)`        | 结果只有一行                   |

因此，原始整体时间复杂度可以表示为：

```text
O(R + N + P)
```

如果存在整表合并，实际运行开销会更大，内存占用也会更高。

## 8. 时间复杂度优化方案

本次优化主要包括三个方面。

第一，只读取必要字段。整体核心用户购买率分析只需要 `user_id` 和 `behavior_type` 两个字段，因此在读取原始行为数据时使用：

```python
usecols = [
    user_col,
    behavior_col
]
```

这样可以减少不必要字段的读取开销和内存占用。

第二，不进行完整行为表与 RFM 表的大表合并。修改后先从 RFM 表中筛选出核心用户，并将核心用户编号转换为集合：

```python
core_user_set = set(core_users)
```

然后在购买行为中使用：

```python
purchase_chunk[user_col].isin(core_user_set)
```

判断购买用户是否为核心用户。这样避免了将完整 `data_min` 与 RFM 表进行合并，减少了运行开销和内存压力。

第三，使用分块读取原始行为表。通过：

```python
pd.read_csv(..., chunksize=config["chunksize"])
```

每次只读取一部分数据，不一次性加载完整 `data_min`。虽然从时间复杂度角度仍然需要扫描全部数据，但内存占用明显降低，程序更适合处理大数据量文件。

## 9. 优化后的时间复杂度分析

优化后的主要步骤如下：

| 步骤                 | 时间复杂度 | 说明                                   |
| -------------------- | ---------- | -------------------------------------- |
| 读取 RFM 用户分层表  | `O(R)`   | 读取用户分层记录                       |
| 筛选核心用户         | `O(R)`   | 判断每个用户是否为核心用户             |
| 生成核心用户集合     | `O(C)`   | 将核心用户编号保存为集合               |
| 分块读取原始行为数据 | `O(N)`   | 所有行为记录仍需扫描一遍               |
| 筛选购买行为         | `O(N)`   | 判断每一行是否为购买行为               |
| 判断是否为核心用户   | `O(P)`   | 只在购买行为中判断是否属于核心用户集合 |
| 计算购买率           | `O(1)`   | 只计算几个汇总指标                     |
| 保存结果             | `O(1)`   | 结果只有一行                           |

因此，优化后的整体时间复杂度为：

```text
O(R + C + N + P)
```

因为：

```text
C <= R
P <= N
```

所以可以简化为：

```text
O(R + N)
```

如果 `R` 明显小于原始行为记录数 `N`，那么整体可以近似理解为：

```text
O(N)
```

也就是说，这个任务已经接近理论最优。因为无论采用什么方法，都必须至少扫描一遍原始行为数据，才能知道全体购买次数和核心用户购买次数。

## 10. 空间复杂度分析

优化后程序没有一次性读取完整 `data_min` 表，而是使用分块读取。程序主要保存以下数据：

| 数据                          | 说明                     |
| ----------------------------- | ------------------------ |
| 核心用户集合`core_user_set` | 保存核心用户编号         |
| 当前分块数据`chunk`         | 当前读取的一部分行为数据 |
| 最终结果表`result`          | 只有一行汇总结果         |

因此空间复杂度主要为：

```text
O(C + B)
```

其中：

```text
C：核心用户数量
B：单个分块大小，即 chunksize
```

由于 `B` 是配置中固定设置的分块大小，例如 500000，因此内存不会随着完整数据量 `N` 线性增长。相比一次性读取完整 `data_min` 表，空间复杂度从接近 `O(N)` 降低为 `O(C + B)`。

## 11. 优化前后对比

| 对比项       | 优化前                                         | 优化后                                             | 优化效果                      |
| ------------ | ---------------------------------------------- | -------------------------------------------------- | ----------------------------- |
| 参数管理     | 路径、字段名、核心用户定义直接写在 notebook 中 | 使用`config.py` 统一管理                         | 修改参数更方便                |
| 代码结构     | 所有逻辑集中在 notebook 中                     | 拆分为配置、核心封装、运行入口                     | 可读性提升                    |
| 接口设计     | 没有统一主接口                                 | 使用`run_overall_core_purchase_analysis(config)` | 调用更简单                    |
| 读取字段     | 可能读取完整行为表所有字段                     | 只读取`user_id` 和 `behavior_type`             | 减少无效读取                  |
| RFM 匹配方式 | 可能进行完整行为表与 RFM 表合并                | 使用核心用户集合和`isin()` 判断                  | 避免大表合并                  |
| 数据读取方式 | 可能一次性读取完整`data_min`                 | 使用`chunksize` 分块读取                         | 降低内存压力                  |
| 时间复杂度   | `O(R + N + P)`                               | `O(R + C + N + P)`，简化为 `O(R + N)`          | 大 O 接近不变，但实际运行更轻 |
| 空间复杂度   | 可能接近`O(N)`                               | `O(C + B)`                                       | 内存占用明显降低              |

## 12. 最终结论

本次修改完成了整体核心用户购买率分析代码的配置参数文件设计、接口定义、代码封装和时间复杂度审查。

在配置参数方面，使用 `config.py` 集中管理输入路径、输出路径、字段名、核心用户定义、购买行为编号、分块读取大小和输出文件名。这样后续修改路径或参数时，不需要改动核心代码。

在代码封装方面，将完整分析流程拆分为 `load_core_users()`、`count_purchase_by_chunks()`、`build_overall_result()`、`save_result()` 和 `run_overall_core_purchase_analysis()` 等函数。每个函数只负责一个步骤，代码结构更加清晰。

在接口定义方面，使用 `run_overall_core_purchase_analysis(config)` 作为统一接口。外部运行文件只需要导入配置并调用该函数，即可完成完整分析流程。

在时间复杂度方面，本任务本身没有排序和复杂分组，核心任务是统计全体购买次数和核心用户购买次数。由于必须扫描原始行为数据才能得到准确结果，因此理论上整体时间复杂度接近 `O(N)`。优化后整体复杂度可以表示为：

```text
O(R + C + N + P)
```

由于 `C <= R` 且 `P <= N`，可以简化为：

```text
O(R + N)
```

如果 RFM 用户数量 `R` 小于原始行为数据量 `N`，则整体可以近似看作：

```text
O(N)
```

在空间复杂度方面，优化前如果一次性读取完整行为表，空间复杂度可能接近 `O(N)`。优化后使用分块读取，只保存核心用户集合和当前分块数据，空间复杂度为：

```text
O(C + B)
```

其中 `B` 为单个分块大小。通过本次修改，虽然理论时间复杂度已经接近最优，无法再明显降低到低于 `O(N)`，但代码的结构、可维护性、运行稳定性和内存效率都得到了明显提升。
