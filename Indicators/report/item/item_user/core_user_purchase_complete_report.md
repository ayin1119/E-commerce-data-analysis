# 商品核心用户购买率分析

## 1. 报告背景

本报告围绕“商品核心用户购买率分析”进行代码规范化改造和时间复杂度优化说明。原始 notebook 的目标是读取原始用户行为数据 `data_min.csv` 和 RFM 用户分层结果表，筛选出 RFM 类型为“重要价值用户”的核心用户，然后统计每个商品的总购买次数、核心用户购买次数、非核心用户购买次数，并计算核心用户购买率。

## 2. 修改目标

本次修改目标不是改变业务指标，而是让代码更符合项目化要求，并尽量减少不必要的时间开销。

| 修改目标       | 说明                                                                                  |
| -------------- | ------------------------------------------------------------------------------------- |
| 配置参数文件   | 将输入路径、输出路径、字段名、核心用户类型、购买行为编号、分块大小等放入`config.py` |
| 接口定义       | 使用`run_core_user_purchase_analysis(config)` 作为统一运行接口                      |
| 代码封装       | 将完整流程拆分为多个函数，每个函数只负责一个步骤                                      |
| 时间复杂度优化 | 默认不进行最终全量排序，避免不必要的`O(G log G)`                                    |
| 内存优化       | 保留分块读取方式，不一次性读取完整`data_min`                                        |
| 可维护性优化   | 后续改路径、改核心用户定义、改输出文件名时，只需要改配置文件                          |

## 3. 修改后的项目结构

修改后建议将项目拆分为三个 Python 文件：

```text
core_user_purchase_project/
│
├── config.py
├── core_user_purchase_pipeline.py
└── run_core_user_purchase.py
```

三个文件的职责如下：

| 文件                               | 作用                                                                                       |
| ---------------------------------- | ------------------------------------------------------------------------------------------ |
| `config.py`                      | 配置参数文件，保存路径、字段名、核心用户类型、购买行为编号、分块大小、是否排序和输出文件名 |
| `core_user_purchase_pipeline.py` | 核心代码封装文件，负责读取核心用户、分块读取购买行为、统计购买次数、计算购买率和保存结果   |
| `run_core_user_purchase.py`      | 运行入口文件，负责导入配置并调用统一接口运行程序                                           |

这种结构和之前 RFM 项目的封装方式一致：配置文件集中保存参数，核心流程文件封装函数，运行入口文件只负责调用主接口。

## 4. 业务处理流程

修改后的完整业务流程如下：

```text
读取配置参数
      ↓
读取 RFM 用户分层结果表
      ↓
筛选 rfm_type = 重要价值用户的核心用户
      ↓
将核心用户 user_id 转成集合
      ↓
分块读取原始行为数据 data_min
      ↓
只保留 behavior_type = 4 的购买行为
      ↓
判断购买用户是否属于核心用户
      ↓
按 item_category + item_id 分组统计购买次数
      ↓
合并所有分块的统计结果
      ↓
计算非核心用户购买次数
      ↓
计算核心用户购买率
      ↓
根据配置决定是否排序
      ↓
保存结果 CSV
```

最终输出结果表字段包括：

| 字段                             | 含义               |
| -------------------------------- | ------------------ |
| `item_category`                | 商品类别           |
| `item_id`                      | 商品编号           |
| `total_purchase_count`         | 商品总购买次数     |
| `core_user_purchase_count`     | 核心用户购买次数   |
| `non_core_user_purchase_count` | 非核心用户购买次数 |
| `core_user_purchase_rate`      | 核心用户购买率     |

其中核心公式为：

```text
核心用户购买率 = 核心用户购买次数 / 商品总购买次数
```

非核心用户购买次数为：

```text
非核心用户购买次数 = 商品总购买次数 - 核心用户购买次数
```

## 5. 时间复杂度优化前分析

设：

```text
N：原始行为数据 data_min 的总行数
P：购买行为数据行数，也就是 behavior_type = 4 的行数，P <= N
G：商品类别 + 商品 ID 的组合数量
C：核心用户数量
```

原始代码虽然已经使用分块读取，但最后对完整结果进行了排序：

```python
item_core_purchase = item_core_purchase.sort_values(
    by=["item_category", "core_user_purchase_rate", "total_purchase_count"],
    ascending=[True, False, False]
)
```

如果商品组合数量为 `G`，完整排序的时间复杂度为：

```text
O(G log G)
```

因此原始做法可以表示为：

```text
读取和扫描原始行为数据：O(N)
筛选购买行为：O(N)
处理购买行为并统计：O(P)
计算购买率：O(G)
最终完整排序：O(G log G)
```

所以原始整体时间复杂度可以写为：

```text
O(C + N + P + G log G)
```

## 6. 时间复杂度优化方案

本次优化主要做了两点。

第一，保留分块读取，不一次性读取完整 `data_min`。这样可以降低内存压力，避免数据量大时程序占用过多内存。

第二，将最终排序设置为可选项，默认不排序：

```python
"sort_result": False
```

当 `sort_result = False` 时，程序不会执行完整排序，避免产生 `O(G log G)` 的时间开销。只有当用户确实需要排序后的展示效果时，才把它改成：

```python
"sort_result": True
```

这样既保留了排序功能，也让默认运行方式更高效。

## 7. 优化后的时间复杂度

优化后，主要流程的复杂度如下：

| 步骤                      | 时间复杂度       | 说明                                     |
| ------------------------- | ---------------- | ---------------------------------------- |
| 读取 RFM 表并筛选核心用户 | `O(C)`         | 需要读取 RFM 表并筛选核心用户            |
| 分块扫描原始行为数据      | `O(N)`         | 虽然分块读取，但所有行为数据仍需扫描一遍 |
| 筛选购买行为              | `O(N)`         | 判断每条行为是否为购买行为               |
| 判断是否核心用户          | `O(P)`         | 只对购买行为判断用户是否在核心用户集合中 |
| 按商品分组统计            | `O(P)`         | 对购买行为按商品类别和商品 ID 统计       |
| 合并分块统计结果          | `O(G)`         | 合并每个商品的累计购买次数               |
| 计算核心用户购买率        | `O(G)`         | 每个商品计算一次购买率                   |
| 默认不排序                | 无`O(G log G)` | 避免不必要的完整排序                     |

因此默认 `sort_result = False` 时，整体时间复杂度为：

```text
O(C + N + P + G)
```

由于购买行为数量 `P` 不会超过总行为数量 `N`，所以可以进一步写成：

```text
O(C + N + G)
```

如果商品组合数量 `G` 相比原始行为数据量 `N` 小很多，那么整体运行趋势可以近似理解为：

```text
O(N)
```

如果配置为 `sort_result = True`，则会增加排序步骤，整体复杂度变为：

```text
O(C + N + P + G log G)
```

因此，本次优化的核心结论是：默认情况下去掉非必要全量排序，使复杂度从 `O(C + N + P + G log G)` 优化为 `O(C + N + P + G)`。

## 8. 空间复杂度分析

优化前如果一次性读取完整 `data_min`，空间复杂度可能接近：

```text
O(N)
```

优化后采用分块读取方式，不一次性保存完整行为表。程序主要保存：

```text
核心用户集合
每个分块的商品统计结果
合并后的商品购买统计表
最终商品核心用户购买率结果表
```

因此空间复杂度主要由核心用户数量和商品组合数量决定，可以写成：

```text
O(C + G)
```

其中，`C` 是核心用户数量，`G` 是商品类别和商品 ID 的组合数量。

## 9. 优化前后对比

| 对比项       | 优化前                                   | 优化后                                                   | 优化效果                   |
| ------------ | ---------------------------------------- | -------------------------------------------------------- | -------------------------- |
| 参数管理     | 路径、字段、输出目录直接写在 notebook 中 | 使用`config.py` 集中管理                               | 后续修改更方便             |
| 代码结构     | 所有逻辑集中在一个代码单元               | 拆分为多个函数                                           | 可读性和复用性提升         |
| 运行方式     | 直接运行 notebook 代码                   | 使用`run_core_user_purchase_analysis(config)` 统一接口 | 调用更清晰                 |
| 大表读取     | 已使用分块读取                           | 保留并规范化分块读取                                     | 降低内存压力               |
| 核心用户判断 | 使用集合判断核心用户                     | 保留并封装为函数                                         | 判断效率高，逻辑清晰       |
| 最终排序     | 默认完整排序                             | 默认不排序，可通过配置开启                               | 避免不必要的`O(G log G)` |
| 时间复杂度   | `O(C + N + P + G log G)`               | `O(C + N + P + G)`                                     | 去掉非必要排序后更接近线性 |
| 空间复杂度   | 如果整表读取可能接近`O(N)`             | `O(C + G)`                                             | 更适合大数据量             |

## 10. 最终结论

本次修改完成了接口定义、代码封装、配置参数文件设计和时间复杂度优化。修改后的项目由 `config.py`、`core_user_purchase_pipeline.py` 和 `run_core_user_purchase.py` 三个文件组成。其中，`config.py` 负责统一管理参数，`core_user_purchase_pipeline.py` 负责核心分析流程，`run_core_user_purchase.py` 负责运行程序。

在时间复杂度方面，原始代码最后对所有商品结果进行了完整排序，整体复杂度可以表示为 `O(C + N + P + G log G)`。优化后将排序设置为可选项，默认不排序，因此避免了 `O(G log G)` 的额外开销，整体复杂度优化为 `O(C + N + P + G)`。由于 `P <= N`，在商品组合数量相对较小的情况下，整体运行趋势可以近似看作 `O(N)`。

在空间复杂度方面，修改后继续采用分块读取方式，不一次性加载完整 `data_min` 表，主要保存核心用户集合和商品购买统计结果，因此空间复杂度为 `O(C + G)`。
